# Home ADB Guard

A small Android app that turns ADB-over-Wi-Fi on with one button, and turns it
back off when the phone leaves a trusted network.

![Build](https://github.com/0x45796164/HomeAdbGuard/actions/workflows/build.yml/badge.svg)

**[Download the latest APK](https://github.com/0x45796164/HomeAdbGuard/releases/latest)**

## The problem

On stock Android, enabling Wireless debugging is a multi-step trip every
time:

1. Settings → About phone → Software information → tap **Build number** 7
   times.
2. Settings → System → Developer options → flip **Wireless debugging** on.

Anything that turns Developer options off — a banking app that refuses to
launch with it on, a screen-sharing prompt, a factory reset, a fresh phone —
clears Wireless debugging with it. The dance starts over. There is no
shortcut, no Quick Settings tile, no notification action: the toggle is
deliberately buried.

## What this app does

Puts that toggle behind one button. Optionally, flips it automatically when
the phone joins or leaves a Wi-Fi network marked as trusted.

That is the whole feature set.

## Use cases

**Local app development.** Pushing debug builds to a phone over the LAN —
ADB-over-Wi-Fi comes up the moment the phone is on the home network, with
no cable and no settings dive.

**Remote desktop via [scrcpy](https://github.com/Genymobile/scrcpy).** `scrcpy`
over Wi-Fi delivers a full Android desktop on any computer that's on the same
network as the phone: drag-and-drop APK installs, drag-and-drop file pushes,
a virtual display so the phone's screen can stay off. The catch is that
Wireless debugging needs to be up, and Android keeps cycling it off. Home ADB
Guard is the glue.

A representative scrcpy invocation:

```sh
scrcpy \
  --new-display=1920x1080/220 \
  --start-app=com.sec.android.app.launcher \
  --video-codec=h264 \
  --video-bit-rate=12M \
  --max-fps=60 \
  --audio-codec=opus \
  --audio-bit-rate=96K \
  --audio-buffer=50 \
  --display-ime-policy=local \
  --keyboard=uhid \
  --mouse=sdk \
  --no-mouse-hover \
  --keep-active
```

Most Android "desktop mode" options are either wired-only or assume an
external display. `scrcpy` over Wi-Fi works from any computer on the LAN, but
every time Wireless debugging trips off it sends the user back through
Developer options. Home ADB Guard exists to remove that friction.

## Why not Tasker / Automate / [insert super-app]

Those work and can express this rule. They also want broad permissions, run
their own automation runtime, and require the user to author and maintain
the automation. A single-purpose APK with no analytics and no network code
is a smaller footprint than a general-purpose automation platform.

## Install

```sh
adb install -r app-release.apk
adb shell pm grant app.homeadbguard android.permission.WRITE_SECURE_SETTINGS
```

A normal APK cannot write `Settings.Global` values — the
`WRITE_SECURE_SETTINGS` grant from a computer is the reason the app needs ADB
to set itself up. Open the app afterwards and follow the prompts.

CI publishes a fresh APK to the [latest release](https://github.com/0x45796164/HomeAdbGuard/releases/latest)
on every push to `main`. It is signed with the dummy keystore committed in
[`ci/keystore/`](ci/keystore/) — a stable sideloadable signature, not a real
release identity. Treat it the way you treat any debug APK.

Full walkthrough: [docs/USAGE.md](docs/USAGE.md).

## What it doesn't do

- No `INTERNET` permission. No cleartext traffic. The app cannot reach the
  network at all.
- No analytics, telemetry, crash reporting, or third-party SDKs.
- Only Google AndroidX UI libraries (`androidx.appcompat`, Material
  Components).
- Java + Views. No Kotlin, no Compose.
- Min Android 14.
