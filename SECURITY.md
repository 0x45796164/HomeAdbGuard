## Threat model

This is a **convenience tool**, not a security boundary. It does not stop a
determined local attacker with physical or network access to the device from
turning ADB-over-Wi-Fi on by other means.

Out of scope:

- An attacker with shell access already.
- An attacker who can spoof the home Wi-Fi's BSSID and SSID.
- A malicious or compromised ROM, kernel, or system app.
- Another app that holds `WRITE_SECURE_SETTINGS` (rare on stock builds).
- Physical access — the user can toggle the underlying settings manually at
  any time.

In scope: **reducing attack surface** when the device leaves a trusted Wi-Fi
network. The app requests Wireless debugging and Developer options off, as
fast as the OS will allow.

## Inter-app surface

The app exposes the minimum surface required to function on Android:

- `MainActivity` is exported (it has `MAIN/LAUNCHER`, which Android requires
  to be exported). It does **not** declare any custom action intent-filter,
  so no other app can implicitly drive its UI to toggle ADB. The only thing
  another app can do via this component is launch the activity — which is
  what any launcher icon already does.
- `MonitorTileService` is the Quick Settings tile. Android requires it to
  hold `android:permission="android.permission.BIND_QUICK_SETTINGS_TILE"`,
  which means only the system process can bind it. No other app can invoke
  the tile.
- All broadcast receivers (`BootReceiver`, `ControlReceiver`,
  `HomeArrivalReceiver`) are `android:exported="false"` and reachable only
  from inside the app's own process.
- The foreground `MonitorService` is `android:exported="false"`.

## Reporting a vulnerability

Open a private security advisory through the repository host
("Report a vulnerability" on GitHub). Don't open a public issue for an
exploitable finding before maintainers have had a chance to respond.

Include: device, Android version, OEM, `versionName`, repro steps, expected
vs. observed behavior, and impact assessment.

There is no SLA. The project is in maintenance mode for security and
bug-fix work only (see [README.md](README.md#disclosures)) — feature work
is not planned.

## Verifying behavior

```sh
adb shell settings get global development_settings_enabled
adb shell settings get global adb_wifi_enabled
adb shell dumpsys package app.homeadbguard | grep -E 'INTERNET|grantedPermissions' -A40
```

The package should never hold `android.permission.INTERNET`.

## Supply chain

Runtime dependencies:

- `androidx.appcompat:appcompat`
- `com.google.android.material:material`

No analytics, no ad libraries, no crash reporters, no network clients. The
only Gradle plugin is `com.android.application`. Any fork should audit
`app/build.gradle.kts` and `settings.gradle.kts` before adding anything.

## CI release signing

The APK published on the releases page is signed with the dummy keystore in
[`ci/keystore/`](ci/keystore/) — passwords are in the repo. It is a
sideloadable signature, not a real release identity, and it is stable across
builds so updates install without uninstall. For a stronger guarantee, build
from source.
